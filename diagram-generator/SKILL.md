# Diagram Generator — 通用图表生成 Skill

> **触发方式**：用户说"画一张架构图""生成流程图""帮我画个关系图"等涉及可视化图表输出的需求时启用。
> **产出**：SVG 源文件 + PNG 高清图片


## 一、适用场景

- 架构图（系统架构、Skill 架构、模块关系）
- 流程图（SOP 流程、审批流程、数据流向）
- 关系图（实体关系、层级结构、组织架构）
- 对比图（方案对比、版本对比）
- 任何需要以图片形式呈现结构化信息的场景


## 二、设计原则

### 2.1 风格默认值（用户未指定时）

| 项目 | 默认值 |
|------|--------|
| 底色 | 浅色（`#FAFBFC`） |
| 字体 | `-apple-system, 'Segoe UI', 'Microsoft YaHei', sans-serif` |
| 配色 | 蓝(`#5B8DEF`)、绿(`#4CAF82`)、红(`#E05D5D`)、紫(`#9B7FD4`)、橙(`#E8923A`) |
| 风格 | 简约、扁平、无过多装饰 |
| 圆角 | 卡片 `rx="7~10"`，胶囊 `rx="18~22"` |
| 署名 | 若用户要求，右下角加粗显示 |

### 2.2 核心设计规则

1. **信息层级清晰**：标题 → 流程/阶段条 → 主体内容 → 产出/总结 → 署名，自上而下
2. **对齐严谨**：所有同层模块必须顶部对齐、等宽或等比分布；文字用 `text-anchor="middle"` 居中
3. **留白克制**：内容紧凑，不留大面积空白；viewBox 高度根据实际内容裁剪
4. **连线简洁**：同层用 `→` 箭头或短线，跨层用竖线+marker；避免交叉、避免过多虚线散射
5. **独立模块突出**：如有"随时介入""可选"等非线性模块，用虚线边框(`stroke-dasharray`)独立放置，一条连线接入主体即可
6. **颜色有意义**：不同模块/角色用不同色条标识，保持全图色彩体系统一


## 三、执行 SOP

### Step 1：需求理解

从用户描述中提取：

- **图表类型**：架构图 / 流程图 / 关系图 / 其他
- **包含模块**：列出所有需要展示的节点/模块
- **层级关系**：谁统筹谁、谁依赖谁、数据流向
- **特殊要求**：配色偏好、署名、尺寸、深色/浅色
- **输出格式**：默认 SVG + PNG，用户可指定仅其一

若信息不足，主动追问：「需要包含哪些模块？层级关系是什么？有配色偏好吗？」

### Step 2：布局规划（先想后画）

在动手写 SVG 前，先在心中规划：

```
1. 确定 viewBox 尺寸（默认 1200×640，内容多则加高）
2. 划分纵向层级（每层高度分配）
3. 划分横向列数（同层模块数量 → 等宽分配 → 计算 x 坐标）
4. 确定连线方案（层间竖线、同层横箭头、独立模块单线连接）
5. 预留底部区域（产出物/总结/署名，紧凑排列）
```

**关键计算公式**：
- 同层 N 个等宽卡片，容器宽度 W，间距 gap：
  - 卡片宽 = `(W - (N-1)*gap) / N`
  - 第 i 个卡片 x = `起始x + i * (卡片宽 + gap)`
  - 卡片中心 = `x + 卡片宽/2`

### Step 3：编写 SVG

遵循以下结构模板：

```xml
<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 {W} {H}" width="{W}" height="{H}">
  <defs>
    <style>text { font-family: ...; }</style>
    <!-- marker 定义（箭头） -->
  </defs>

  <!-- 1. 背景 -->
  <rect width="{W}" height="{H}" fill="#FAFBFC"/>

  <!-- 2. 标题 + 副标题 -->

  <!-- 3. 流程/阶段条（如有） -->

  <!-- 4. 主体层级（统筹层 → 执行层 等） -->

  <!-- 5. 连线与箭头 -->

  <!-- 6. 产出物/总结区 -->

  <!-- 7. 底部版本 + 署名 -->
</svg>
```

**SVG 编写规范**：
- 所有文字用 `<text>` 而非 `<foreignObject>`（兼容性好）
- 卡片 = `<rect>` 背景 + `<rect>` 顶部色条(height=3.5) + `<text>` 标题/描述
- 箭头用 `<marker>` + `<line>` 或 `<path>`
- 虚线用 `stroke-dasharray="5,3"` 或 `"6,3"`
- emoji 可直接写入 `<text>`（sharp 渲染支持有限，优先用文字符号如 ✓ ✗ ↗ ↻）

### Step 4：SVG 转 PNG

**环境要求**：Node.js + `sharp` 库（已安装则跳过 install）

```javascript
const sharp = require('sharp');
const fs = require('fs');
const svgBuf = fs.readFileSync('{SVG路径}');
sharp(svgBuf, { density: 200 })
  .resize({输出宽度}, {输出高度})
  .png()
  .toFile('{PNG路径}')
  .then(info => console.log('Done', JSON.stringify(info)))
  .catch(err => console.error('Error', err));
```

**尺寸映射**：
| SVG viewBox | PNG 输出 | 适用场景 |
|-------------|----------|----------|
| 1200×640 | 2400×1280 | 标准横版（汇报用） |
| 1200×720 | 2400×1440 | 内容较多 |
| 1200×900 | 2400×1800 | 大型架构 |
| 800×600 | 1600×1200 | 小型图 |

**注意**：
- 写临时脚本执行，执行完毕后**必须删除**临时 js 文件
- 若 sharp 未安装，先在工作目录执行 `npm install sharp`
- Python 环境不可用时，一律用 Node.js + sharp 方案

### Step 5：验证与迭代

1. 用 `read_file` 查看 PNG 效果
2. 检查：对齐是否准确、文字是否溢出、留白是否合理、署名是否到位
3. 用户反馈 → 针对性修改 SVG → 重新转 PNG
4. 常见修改方向：
   - "太花" → 减少渐变/阴影，用纯色+细边框
   - "不对齐" → 重新计算 x/y 坐标，确保同层模块 y 值相同
   - "留白多" → 缩小 viewBox 高度，压缩层间距
   - "箭头乱" → 减少连线数量，一条线代替多条散射线
   - "不清晰" → 加大 font-size，加粗关键文字


## 四、避坑清单

| 问题 | 解决方案 |
|------|----------|
| SVG 文字在 PNG 中模糊 | density 设为 200+，输出尺寸 ≥ viewBox 的 2 倍 |
| emoji 在 sharp 中渲染异常 | 改用文字符号（✓ ✗ → ↻）或去掉 |
| 同层模块不对齐 | 用公式计算坐标，不手动估算 |
| 虚线箭头太多导致视觉混乱 | 同类连接合并为一条，或用标注文字替代 |
| 底部大面积空白 | viewBox 高度 = 最后一个元素 y + padding(30~40) |
| Python 不可用 | 一律用 Node.js + sharp，不尝试 Python |
| 命令超长 | 写入临时 .js 文件执行，执行后删除 |
| read_file 无法读 SVG 内容 | 用 `type` 命令（Win）或 `cat` 命令（Mac/Linux）读取 |
